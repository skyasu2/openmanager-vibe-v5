# Extended Thinking 기능 검증 - AI 교차검증 의사결정

**날짜**: 2025-11-04
**상황**: Claude Code의 Extended Thinking 기능에 대한 공식/비공식 정보를 구분하고, Phase 2 문서에 포함할 내용을 결정하기 위한 3-AI 교차검증

---

## 🤖 AI 의견 요약

### 📊 Codex (실무 관점) - 19초, 3,772 토큰

- **핵심 주장**: 공식 문서 정보는 신뢰 가능하며, 비공식 정보(ultrathink, 토큰 숫자, 무지개 색)는 유지보수 리스크가 크므로 문서에서 제외해야 함
- **근거**:
  - 공식 문서의 키워드(Toggle, MAX_THINKING_TOKENS, 프롬프트 강조)는 재현 가능한 동작
  - 비공식 키워드는 제품 업데이트로 무력화되거나 예외 발생 가능
  - 출처 불명 정보는 문서화 시 오해 유발 위험
- **추천 사항**:
  - 포함: 공식 키워드, 활성화 방법, 기본 상태, 시각 피드백, 비용 여부
  - 제외: 비공식 키워드·숫자·효과, 실험 버전 UI, 커뮤니티 루머
  - 실무 테스트: 공식 토글과 MAX_THINKING_TOKENS 테스트로 ROI 확인

### 📐 Gemini (아키텍처 관점) - 44초

- **핵심 주장**: 문서 구조는 매우 적절하나, 공식 문서 자체가 SSOT 원칙을 위반하고 있음 (ultrathink 같은 비공식 정보가 커뮤니티에 파편화)
- **근거**:
  - 공식/비공식 정보 분리가 명확 (출처 URL, 명시적 표기)
  - 정보 계층 우수 (키워드, 활성화 방법 등 체계적 구조)
  - 하지만 비공식 정보가 공식 문서에 누락되어 SSOT 위반
- **추천 사항**:
  - 이 문서가 프로젝트 내 SSOT 역할 수행
  - 정보 파편화 문제 해결 시도는 좋은 방향
  - 공식 문서 개선 필요 (장기 과제)

### ⚡ Qwen (성능 관점) - 68초

- **핵심 주장**: Extended Thinking은 2-15배 응답 시간 증가, 토큰 예산과 응답 시간은 선형 비례, 복잡한 작업에만 선택적 사용 권장
- **근거**:
  - 중간 추론 단계 생성으로 계산 비용 2-3배 증가
  - 토큰 소비 비례 증가 (thinking + response)
  - 메모리 사용량 증가
- **추천 사항**:
  - 토큰 예산: 4K (기본), 10K (복잡), 32K (최대)
  - 선택적 활성화 (20% 인터랙션만)
  - 간단한 작업에는 비활성화 (토큰 절약)

## ⚖️ 합의점과 충돌점

### ✅ 합의

- 공식 문서 정보는 신뢰 가능, 문서에 포함
- 비공식 정보(ultrathink, 무지개 색)는 위험, 문서에서 제외
- 공식/비공식 명확 분리 필요

### ⚠️ 충돌

#### 1. SSOT 평가

- **Codex**: 문서 제외 기준 명확, 실무적 접근
- **Gemini**: SSOT 원칙 위반 지적 (공식 문서 자체가 불완전)
- **Qwen**: 성능 영향 중심 분석 (SSOT 언급 없음)

**해석**: Gemini의 지적은 정당하나, 공식 문서 개선은 우리 통제 밖. 현실적으로 프로젝트 내 SSOT 역할은 충분히 수행 가능.

#### 2. 토큰 숫자 처리

- **Codex**: 완전 제외 (유지보수 리스크, 공식 근거 없음)
- **Gemini**: 언급 없음
- **Qwen**: 성능 분석 기준으로 사용 (4K, 10K, 32K)

**해석**: Qwen의 토큰 숫자는 성능 분석에는 유용하나, 공식 문서 근거가 없어 문서화는 부적절. Codex 의견 채택.

## 🎯 최종 결정

**채택된 방안**: 공식 정보만 문서화, 비공식 정보 제외, 성능 지침 추가

**근거**:

- **신뢰성**: 공식 문서 근거만 사용 → 유지보수 가능
- **안정성**: 비공식 정보 제외 → 오해 방지, 업데이트 리스크 회피
- **실용성**: 성능 지침 추가 → 1인 개발자 ROI 중심
- **프로젝트 컨텍스트**: Phase 2 문서는 신뢰 가능한 워크플로우 가이드 목적

**기각된 의견**:

- **Gemini의 "SSOT 원칙 위반" 지적**: 공식 문서 개선은 Anthropic의 책임, 우리 통제 밖. 프로젝트 내 SSOT 역할은 충분히 수행 가능
- **Qwen의 토큰 숫자 사용**: 성능 분석은 유용하지만 공식 근거 없어 문서화 부적절. 대신 "처리 시간 증가 가능성" 등 일반적 표현 사용
